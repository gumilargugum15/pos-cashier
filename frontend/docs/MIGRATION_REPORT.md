## 6. Deployment instructions (Nginx + Laravel API)

The frontend is now a **static asset bundle** (`dist/`) with **no Node.js
server required at runtime** — Nitro is gone, there is nothing to run. Deploy
`dist/` as static files behind Nginx, proxying API calls to the existing
Laravel backend.

### 6.1 Build

```bash
cd frontend
npm ci
VITE_API_BASE_URL=https://api.yourdomain.com/api/v1 npm run build
# → produces frontend/dist/
```

Copy `dist/` to the server, e.g. `/var/www/kagoem-pos/frontend/dist`.

### 6.2 Nginx server block

```nginx
server {
    listen 80;
    server_name pos.yourdomain.com;

    root /var/www/kagoem-pos/frontend/dist;
    index index.html;

    # Long-cache hashed build assets
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # SPA fallback: any unknown path resolves to index.html so
    # TanStack Router can handle it client-side (e.g. a hard reload on /settings).
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Optional: reverse-proxy the Laravel API under the same origin to avoid CORS.
    location /api/ {
        proxy_pass http://127.0.0.1:8000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

If the API is proxied under the same origin as above, build with
`VITE_API_BASE_URL=/api/v1` instead of an absolute URL, and ensure Laravel's
`APP_URL`/CORS config (`config/cors.php`) allows the frontend origin.

If the API stays on a separate origin/subdomain instead, keep
`VITE_API_BASE_URL` absolute (as in §6.1) and make sure Laravel's
`config/cors.php` `allowed_origins` includes the frontend's domain — this
project's Axios client already sends `Authorization: Bearer <token>` headers
and no cookies, so `supports_credentials` does not need to be enabled.

### 6.3 Laravel API — no changes required

The backend (`backend/`) is untouched by this migration. It continues serving
`/api/v1/*` exactly as before; Sanctum token auth, all controllers, and the
database are unaffected. No backend deployment steps change.

### 6.4 HTTPS

Terminate TLS at Nginx (Let's Encrypt/Certbot or your existing cert pipeline)
in front of the server block above; no application-level changes are needed
either side.

## 7. Rollback steps

A local git repository was initialized specifically for this migration, with
a checkpoint commit (`"Checkpoint before TanStack Start -> Vite SPA
migration"`) capturing the exact pre-migration state before any file was
touched.

**If the migration needs to be fully reverted:**

```bash
cd /var/www/kagoem-pos/frontend
git log --oneline               # confirm the checkpoint commit is the one you expect
git reset --hard <checkpoint-commit-sha>
rm -rf node_modules dist
npm install
```

This restores `@lovable.dev/vite-tanstack-config`, `@tanstack/react-start`,
`nitro`, `src/server.ts`, `src/start.ts`, the two SSR error-handling lib
files, `bun.lock`/`bunfig.toml`, and the original `vite.config.ts`/
`__root.tsx`/`package.json`/`eslint.config.js`/`.gitignore` — i.e. the exact
TanStack Start + Nitro setup that existed before this migration began.

**If only a specific piece needs reverting** (e.g. keep the SPA migration but
restore one behavior), use `git diff <checkpoint-commit-sha> -- <path>` to see
the exact pre-migration version of that file and cherry-pick the change
manually rather than a full reset.

**No backend rollback is needed under any scenario** — the Laravel API was
never modified by this migration.
